# 机器人任务执行 — 前端对接指南

## 整体流程

```
创建任务 → 下发执行 → 查看结果
```

前端需要对接的关于执行的接口只有 **1 个**：POST /api/robot/{id}/dispatch?taskId=xx

具体来说：

```
① 调 POST /api/task          创建一个任务（绑定流程+机器人），状态为 PENDING
② 调 POST /api/robot/{id}/dispatch?taskId=xx   下发给机器人执行
③ 用户在任务列表刷新，看到状态变成 SUCCESS 后点进去看结果
```

## 分步说明

### 第一步：创建任务

```
POST /api/task
```

```json
{
  "taskName": "测试任务-采集管理员信息",
  "processId": 100,
  "robotId": 100
}
```

返回中拿到 `data.id`，这就是后面的 `taskId`。

> 任务的增删改查都是 `/api/task` 下的标准 REST 接口，不再赘述。

### 第二步：下发执行

```
POST /api/robot/{robotId}/dispatch?taskId={taskId}
```

- `robotId`：路径参数，机器人 ID
- `taskId`：查询参数，第一步创建的任务 ID

返回：`{ "code": 200, "data": "任务已提交" }`

调用后接口**立即返回**，任务在后台线程池异步执行。前端可以弹个提示"任务已提交，执行中..."。

可能报错：
- 404：机器人或任务不存在
- 409：机器人不是 online 状态，或任务不是 PENDING/FAILED 状态

### 第三步：查看结果

用户在任务列表页刷新，看到任务状态变化：

| 任务状态 | 含义 |
|---------|------|
| `PENDING` | 待执行，可dispatch |
| `RUNNING` | 执行中 |
| `SUCCESS` | 成功，点进去看结果 |
| `FAILED` | 失败，可重新 dispatch |

点击任务进入详情，关注返回的三个字段：

```
GET /api/task/{taskId}
```

| 字段 | 说明 |
|------|------|
| `status` | `SUCCESS` 或 `FAILED` |
| `remark` | 执行摘要，如 `[采集] 登录成功\n[解析] 查询到3条记录` |
| `resultData` | Python 返回的原始数据，如 `姓名: 系统管理员\n邮箱: xxx\n手机号: xxx` |

## 状态流转

### 任务状态

```
PENDING ──dispatch──▶ RUNNING ──▶ SUCCESS（成功）
    ▲                    │
    │                    └──▶ FAILED（失败，可重新 dispatch）
    │
  新创建 / 重新执行
```

### 机器人状态

```
online ──dispatch──▶ pending(排队) ──▶ working(执行) ──▶ online(完成)
```
